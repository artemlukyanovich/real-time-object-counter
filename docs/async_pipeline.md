# Асинхронный видеопайплайн (Async pipeline)

Вынос декодирования кадров в фоновый поток, чтобы декод шёл **параллельно** с
инференсом, а не лежал на критическом пути главного цикла. Снимает IO-bound
ограничение и, для живого источника, контролирует задержку через сброс
устаревших кадров.

Управляется двумя параметрами: `video.async_pipeline` (вкл/выкл, по умолчанию
`true`) и `video.drop_policy` (политика сброса кадров).

---

## Зачем это нужно

В синхронном цикле один кадр стоит `декод + инференс` — пока декодируется
следующий кадр, GPU простаивает, и наоборот. Фоновый поток-продьюсер декодирует
кадры заранее, поэтому время кадра падает к `max(декод, инференс)`:

```
sync:   [декод][   инференс   ][декод][   инференс   ] ...
async:  [декод][декод][декод] ...   (фоном)
                 [   инференс   ][   инференс   ] ...
```

Особенно полезно в связке с прореживанием кадров (`detect_interval`, см.
`docs/frame_skipping.md`): на пропущенных кадрах цикл дешёвый
(экстраполяция+рендер), и декод легко становится узким местом — async его
убирает.

> **Почему поток, а не процесс (GIL).** `cap.read()`, `cv2.resize` и
> torch/CUDA-инференс отпускают GIL на время C/CUDA-операций, поэтому обычный
> `threading.Thread`-продьюсер даёт реальное перекрытие декода с инференсом.
> `multiprocessing` не нужен и был бы дороже из-за сериализации кадров.

---

## Как это работает

```
[поток-продьюсер]  cap.read() → resize → queue.put()
        │  (sentinel в конце потока / при ошибке)
        ▼
[главный цикл]  queue.get() → tracker → counter → render → imshow
```

Реализовано в `src/video_source.py`. Сигнатура `read() → (ret, frame)` **не
изменилась**, поэтому главный цикл (`ObjectCounterApp.run`) почти не затронут —
он просто читает кадры как и раньше, не зная, синхронный это режим или нет.

- **Продьюсер** (`_reader_loop`): в цикле декодирует и ресайзит кадры, кладёт в
  `queue.Queue`. В конце потока (или при исключении декода) кладёт sentinel.
- **Потребитель** (`read`): достаёт кадр из очереди; на sentinel возвращает
  `(False, None)`, а если в потоке было исключение — пробрасывает его в главный
  цикл.
- **Завершение** (`release`): выставляет stop-событие, разблокирует возможный
  заблокированный `put`, делает `join` потока и освобождает capture.

## Политика сброса кадров (`drop_policy`)

Главный вопрос асинхронного режима — **что делать, когда инференс не успевает за
потоком кадров**:

| Политика | Поведение | Когда |
|----------|-----------|-------|
| `block` | продьюсер ждёт, пока освободится место в очереди (backpressure). Ни один кадр не теряется, результат детерминированный, но задержка растёт | **видеофайл** |
| `drop` | очередь размером 1; при заполнении старый кадр выбрасывается, обрабатывается самый свежий. Низкая задержка, нагрузка сбрасывается сама, но часть кадров пропускается | **живой поток** (камера/дрон) |
| `auto` | `block` для видеофайлов, `drop` для живых источников | по умолчанию |

### Почему политика разная для файла и потока

- **Видеофайл** — это, по сути, оффлайн-задача: нужно обработать *каждый* кадр.
  Дроп изменил бы подсчёт и сделал бы результат недетерминированным
  (зависящим от загрузки CPU/таймингов). Поэтому файл → `block` (без потерь,
  воспроизводимо). Выигрыш здесь — перекрытие декода с инференсом, без потери
  кадров.
- **Живой поток (камера/дрон)** — real-time задача: если не успеваем, надо
  выбрасывать устаревшие кадры и держать свежий. Для дрона это критично — старый
  кадр означает, что объект уже сместился. На слабом железе `drop` — это и есть
  механизм «не захлебнуться»: система автоматически прореживает поток ровно
  настолько, насколько не тянет.

### Override

`drop_policy` можно задать явно (`block` / `drop`), переопределив авто-выбор.
Основной сценарий — **нагрузочное тестирование edge-кейсов**: прогнать записанный
файл в режиме `drop`, имитируя живой поток с дрона, прямо на целевом слабом
устройстве — без реального дрона. Так видно, сколько кадров система реально
успевает обработать и как держит свежесть под нагрузкой.

---

## Принятые решения

1. **Тумблер вместо «всегда вкл».** Асинхронное чтение почти всегда выгодно, но
   флаг `async_pipeline` оставлен сознательно: (а) чтобы **A/B-замерить** выигрыш
   на одном и том же видео (sync vs async), (б) как fallback при отладке гонок
   потоков. Дефолт — `true`.
2. **Drop-политика по типу источника, а не «один дроп на всё».** Единый дроп
   молча терял бы кадры на файле → регрессия подсчёта и потеря воспроизводимости.
   Поэтому `auto` разводит поведение, а override оставлен для нагрузочных тестов.
3. **Поток, а не процесс** — см. врезку про GIL выше.
4. **Сигнатура `read()` сохранена** — главный цикл не знает о режиме, async
   включается одним флагом без переписывания пайплайна.
5. **Корректное завершение и проброс ошибок** — sentinel в конце потока, проброс
   исключений декода в главный цикл, остановка с `join` в `release()`.

## Конфигурация и запуск

```yaml
video:
  async_pipeline: true   # false = синхронный однопоточный режим
  drop_policy: "auto"    # auto | block | drop
```

```bash
python -m src.main --source data/input/video.mp4
```

При старте фактический режим виден в логе:
`Video: async pipeline ON | drop_policy=auto (block/no-drop)`.

## Как оценить результат

В отличие от ONNX/FP16/прореживания, где выигрыш виден по смене бэкенда, здесь
сравнение — это sync vs async на **одном и том же** видео. Главный индикатор —
**I/O-wait**: время, которое главный цикл простаивает в ожидании кадра.
Печатается в финальной сводке как `avg_io_wait_ms` (метрика `record_io_wait` в
`PerformanceMetrics`).

| Метрика | sync | async (ожидаемо) |
|---------|------|------------------|
| `avg_io_wait_ms` | ≈ время декода (на критическом пути) | → ~0, если декод быстрее инференса |
| `fps` (throughput) | базовый | выше; потолок — время кадра `max(декод, инференс)` |

### Методика A/B (на файле — `block`, детерминизм)

```bash
# async ON (дефолт)
python -m src.main --source data/input/video.mp4

# async OFF: video.async_pipeline: false, затем тот же запуск
python -m src.main --source data/input/video.mp4
```

Сравниваем `avg_io_wait_ms` и `fps` в финальной сводке. Падение `avg_io_wait_ms`
к ~0 — прямое доказательство, что декод ушёл с критического пути.

### Нагрузочный тест живого потока

```yaml
video:
  drop_policy: "drop"   # файл как «живой поток»
```

Покажет, сколько кадров система реально успевает обработать на данном железе и
насколько свежим остаётся кадр (актуально для дрона на слабом edge-устройстве).

---

## Влияние на результат

| Аспект | Эффект |
|--------|--------|
| Throughput (FPS) | растёт, если декод был на критическом пути; потолок — `max(декод, инференс)` |
| I/O-wait | падает к ~0 при `block`, когда декод быстрее инференса |
| Задержка (камера, `drop`) | низкая и стабильная — обрабатывается всегда свежий кадр |
| Подсчёт на файле (`block`/`auto`) | без изменений — обрабатываются все кадры, детерминированно |
| Подсчёт при `drop` | часть кадров пропускается → возможны пропуски событий; режим только для live/нагрузочных тестов |

> **Взаимодействие с `detect_interval`.** При `drop` реальный интервал между
> обработанными кадрами становится нерегулярным (зависит от того, сколько кадров
> отброшено), из-за чего оценка скорости для экстраполяции боксов может слегка
> искажаться. На `block`/`auto` для файла этого нет — кадры идут подряд.
