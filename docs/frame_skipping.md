# Прореживание кадров (Frame skipping)

Снижение нагрузки на инференс за счёт запуска тяжёлого прохода детекция+трекинг
не на каждом кадре, а раз в N кадров. На промежуточных кадрах боксы треков
экстраполируются, поэтому вывод остаётся плавным, а подсчёт — корректным.

Управляется одним параметром — `detector.detect_interval` (по умолчанию `1` =
выключено, полная обратная совместимость).

---

## Зачем это нужно

Инференс (`model.track`) — доминирующая стоимость кадра. Если гонять его раз в N
кадров, нагрузка на GPU/CPU падает примерно в N раз:

| `detect_interval` | Инференсов на 100 кадров | Нагрузка на инференс |
|-------------------|--------------------------|----------------------|
| `1` (выкл.)       | 100                      | базовая              |
| `2`               | 50                       | ~−50%                |
| `3`               | 34                       | ~−66%                |
| `4`               | 25                       | ~−75%                |

Это основной рычаг ускорения там, где модель — узкое место: слабый GPU, CPU-путь
(`.onnx`), крупные модели (`yolov8m`/`l`/`x`). Дополняет другие оптимизации —
FP16 (`half`) и экспорт в ONNX/TensorRT (см. `docs/export_optimization.md`).

> Это **не** механизм real-time-сброса устаревших кадров с живой камеры
> (frame dropping для контроля задержки) — то отдельная задача уровня
> асинхронного пайплайна. Здесь речь о детерминированном прореживании: каждый
> кадр читается и выводится, но детектируется только каждый N-й.

---

## Как это работает

### Пер-кадровая логика (`src/main.py` → `ObjectCounterApp.run`)

Кадр читается **каждый** раз; дальше развилка по `frame_idx % detect_interval`:

**Инференс-кадр** (`_process_inference_frame`, каждый N-й):
1. `tracker.update(frame)` — полный проход детекция+трекинг.
2. Для каждого трека оценивается скорость центроида (смещение с прошлого
   инференс-кадра, делённое на `detect_interval` → пиксели/кадр).
3. ReID (если включён) — по тем же правилам `reid.update_interval`.
4. `counter.update(tracked_objects)` — подсчёт по реальным боксам.
5. Результаты (детекции, треки, counts, скорости, object_id) кешируются.

**Пропущенный кадр** (`_process_skipped_frame`, остальные):
1. Боксы треков из кеша сдвигаются на `velocity × k`, где `k = frame_idx % N`
   (линейная экстраполяция вперёд).
2. Подсчёт и ReID **не** пересчитываются — переиспользуются кешированные
   значения.
3. Слой сырых детекций берётся из кеша как есть (заморожен).

Рендер и `cv2.imshow` выполняются **на каждом** кадре — картинка остаётся
плавной по частоте вывода.

---

## Принятые решения и их обоснование

### 1. Прореживание входного потока, а не отдельный лёгкий трекер

Классический приём «тяжёлый детектор раз в N кадров + лёгкий трекер между ними»
здесь неприменим напрямую: в проекте **детекция и трекинг слиты в один вызов**
`model.track(persist=True)` (`tracker.py`). Ultralytics не предоставляет режим
«трекать без детекции». Завести отдельный Kalman/optical-flow трекер для
промежуточных кадров — это большая отдельная переработка. Поэтому реализовано
прореживание всего прохода с экстраполяцией боксов на стороне приложения.

### 2. Линейная экстраполяция вместо заморозки боксов

На пропущенных кадрах боксы треков двигаются по оценённой скорости, а не стоят на
месте. Это:
- даёт плавную картинку (нет «дёрганья» раз в N кадров);
- точнее по геометрии для пересечения линий и зон при визуальной проверке.

Скорость оценивается по двум последним инференс-кадрам и нормируется на
`detect_interval` (смещение за N реальных кадров → пиксели на кадр). Компромисс:
при резкой смене направления экстраполированный бокс может ненадолго «уехать» от
объекта — это исправляется на следующем инференс-кадре.

### 3. Подсчёт и ReID — только на инференс-кадрах

`counter.update()` вызывается лишь когда есть **наблюдённые** (а не предсказанные)
боксы. Причины:
- предсказанные центроиды не должны порождать фантомные пересечения линий;
- исключается двойной счёт (на замороженных/экстраполированных боксах объект
  «не входит» в зону повторно).

Это безопасно для подсчёта пересечений: счётчик фиксирует смену стороны линии
между сэмплами (`counter.py`), и пока объект попадает в выборку по обе стороны
линии — пересечение засчитывается один раз. Риск только в пропуске события, если
объект успевает пересечь линию целиком *между* двумя инференс-кадрами (растёт с N
и скоростью объекта). ReID аналогично гоняет эмбеддинги только по свежим кропам;
между кадрами переиспользуются последние `object_id`.

### 4. Слой сырых детекций замораживается

У сырых детекций (`show_detections`) нет `track_id`, поэтому привязать к ним
скорость нельзя — они переиспользуются из кеша без сдвига. Боксы треков (с ID, по
ним идёт подсчёт) при этом двигаются. При больших N замороженный слой детекций
может визуально расходиться с треками; это отладочный слой, для чистой картинки
его можно отключить (`display.show_detections: false`).

### 5. `lost_track_buffer` делится на N

`lost_track_buffer` маппится в `track_buffer` Ultralytics и измеряется **в тиках
трекера**, а трекер тикает только на инференс-кадрах. Один тик теперь = N
реальных кадров. Если значение не масштабировать, фактическая длительность буфера
вырастет в N раз (заданные `auto_lost_track_buffer_seconds: 3.0` при N=3
превратились бы в ~9 с реального времени → мёртвые треки висят слишком долго,
выше риск перехвата ID).

Решение: трактовать конфигурное значение как «кадры исходного видео» и делить на
N — `max(1, round(buffer / detect_interval))` — **и для авто-, и для явно
заданного** значения (для консистентности: одно и то же число секунд буфера не
должно вести себя по-разному в зависимости от способа задания). Длительность
буфера в секундах остаётся постоянной при любом N. Реализовано одним местом в
`_initialize_components` после `_resolve_lost_track_buffer`.

> Важно: само *качество ассоциации* при росте N всё равно деградирует (Kalman
> предсказывает на больший шаг dt, меньше перекрытия соседних наблюдений → выше
> риск ID-switch). Это буфером не лечится — это плата за прореживание.

### 6. Две метрики FPS

При `detect_interval > 1` показывать один FPS вводит в заблуждение. В оверлее
выводятся:
- **`FPS`** — throughput, выводимых кадров в секунду (включая дешёвые
  пропущенные кадры);
- **`infer`** — реальная скорость модели, инференсов в секунду (по времени,
  проведённому внутри `model.track`), отражает истинную стоимость инференса
  независимо от прореживания.

Вторая строка появляется только при `detect_interval > 1`. В финальной сводке
(`metrics.get_summary`) добавлен ключ `inference_fps`.

## Конфигурация и запуск

```yaml
detector:
  detect_interval: 3   # детекция каждый 3-й кадр; 1 = выключено
```

```bash
# через конфиг
python -m src.main --config configs/experiments/01_bytetrack_fast.yaml --source data/input/video.mp4
```

При старте фактическое значение видно в логе трекера:
`Tracker: bytetrack (Ultralytics) | fps=30, ..., detect_interval=3`.

## Влияние на результат

| Аспект | Эффект |
|--------|--------|
| Нагрузка на инференс | падает ~в N раз — основной выигрыш |
| Throughput (FPS вывода) | растёт, если узкое место — инференс; ограничен IO/декодом/рендером |
| Трекинг | выше риск ID-switch на быстром движении/окклюзиях при большом N |
| Подсчёт пересечения линий | корректен, пока объект сэмплируется по обе стороны линии; риск пропуска быстрых пересечений между инференс-кадрами |
| Зонный подсчёт | устойчив (фиксируется один раз по факту входа) |
| Визуально | боксы треков плавные (экстраполяция); слой сырых детекций может отставать |

**Рекомендации по выбору N:**
- `2–3` — безопасный диапазон для большинства сцен, заметный выигрыш при малой
  потере качества.
- `4+` — только для медленных сцен / when инференс критично дорог; проверяйте
  подсчёт и стабильность ID на своём видео.
- Если инференс не узкое место (значимую долю кадра занимают ReID/рендер/IO) —
  выигрыш по итоговому FPS будет меньше; рассмотрите сначала FP16/экспорт.

## Связанные оптимизации и задачи на будущее

- **FP16 / экспорт ONNX/TensorRT** — `docs/export_optimization.md`. Комбинируются
  с прореживанием (ускоряют сам инференс, а не его частоту).
- **Асинхронный видеопайплайн** — отдельная задача: вынести чтение/декодирование
  кадров в поток-продьюсер и добавить real-time-сброс устаревших кадров для живой
  камеры. Прореживание снижает *вычислительную* нагрузку, async-пайплайн снимает
  *IO-bound* ограничение и контролирует задержку.
