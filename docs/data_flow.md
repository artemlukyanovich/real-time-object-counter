# Поток данных (Data Flow)

---

## Общая схема pipeline

```
┌─────────────┐    np.ndarray     ┌──────────────┐    List[Detection]   ┌─────────────┐
│ VideoSource │ ────────────────► │   Detector   │ ──────────────────► │   Tracker   │
└─────────────┘   (BGR, HxWx3)   └──────────────┘                      └─────────────┘
                                                                               │
                                                                    Dict[int, (bbox, class)]
                                                                               │
┌─────────────┐    np.ndarray     ┌──────────────┐    Dict[str, int]   ┌─────▼───────┐
│   Display   │ ◄──────────────── │   Renderer   │ ◄────────────────── │   Counter   │
└─────────────┘   (аннотации)    └──────────────┘                      └─────────────┘
```

---

## Этап 1: VideoSource → кадр

**Что происходит:**
- `cv2.VideoCapture.read()` захватывает сырой кадр
- Если разрешение источника ≠ заданному — выполняется `cv2.resize()`
- Счётчик `frame_count` инкрементируется

**Формат выходных данных:**
```python
frame: np.ndarray  # shape: (height, width, 3), dtype: uint8, BGR
# Пример: (720, 1280, 3)
```

**Критичные моменты:**
- При `read()` → `(False, None)` — конец видеофайла или потеря камеры; цикл завершается
- Ресайз выполняется каждый кадр — при слабом CPU может давать просадку FPS на высоких разрешениях

---

## Этап 2: Detector → список детекций

**Что происходит:**
- YOLOv8 выполняет инференс на кадре
- Фильтрация по `confidence_threshold`
- Для каждого обнаруженного объекта: извлечение bbox в формате `xyxy`, имя класса, уверенность

**Формат выходных данных:**
```python
detections: List[Tuple[Tuple[int, int, int, int], str, float]]
# [
#   ((120, 45, 280, 310), "person", 0.87),
#   ((400, 200, 650, 450), "car", 0.92),
# ]
```

- `(x1, y1, x2, y2)` — абсолютные пиксельные координаты (left-top, right-bottom)
- `class_name` — строка из словаря COCO (80 классов)
- `confidence` — float [0.0, 1.0]

**Критичные моменты:**
- Если кадр пришёл пустым (`None`) — `detect()` может упасть; защита на стороне `main.py`
- При `confidence_threshold` слишком высоком → пропуск реальных объектов; слишком низком → шумовые детекции засоряют трекер
- Время инференса YOLOv8n на GPU: ~5–15 мс; на CPU: ~50–200 мс

---

## Этап 3: Tracker → треки объектов

**Что происходит:**
1. Вычисляются центроиды всех новых детекций: `cx = (x1+x2)/2`, `cy = (y1+y2)/2`
2. Для каждого существующего трека ищется ближайшая детекция (евклидово расстояние)
3. Сопоставление: если `distance < max_distance` — трек обновляется
4. Непривязанные детекции регистрируются как новые треки с новым `track_id`
5. Треки без совпадений: `disappeared[id] += 1`; если `disappeared[id] > max_disappeared` — трек удаляется

**Формат выходных данных:**
```python
tracked_objects: Dict[int, Tuple[Tuple[int, int, int, int], str]]
# {
#   0: ((120, 45, 280, 310), "person"),
#   1: ((400, 200, 650, 450), "car"),
#   3: ((50, 100, 180, 250), "person"),
# }
```

- Ключ — уникальный `track_id` (монотонно возрастающий int)
- Значение — текущий bbox и класс объекта

**Критичные моменты:**

- **Потеря треков:** при резком движении объекта центроид смещается более чем на `max_distance` пикселей → новый ID; старый удаляется через `max_disappeared` кадров. Влияет на точность подсчёта.
- **Смена ID при окклюзии:** два объекта, пересекающиеся в кадре, могут поменяться ID — алгоритм не различает их до разделения.
- **Класс объекта:** берётся из последней детекции; если детектор начинает классифицировать объект по-другому (нестабильные предсказания), класс в треке изменится.
- **Параметр `max_distance`** нужно калибровать под реальную скорость объектов и FPS. При 30 FPS объект со скоростью 100 px/s смещается ~3 px/кадр — `max_distance=50` с запасом. При 10 FPS — уже ~10 px/кадр.

---

## Этап 4: Counter → счётчики по классам

**Что происходит:**
1. Для каждого `track_id` в `tracked_objects` проверяется: уже считался ли он (`id in counted_ids`)?
2. Если нет — вычисляется центроид bbox
3. Если задана `count_zone` — проверяется принадлежность центроида полигону (ray casting)
4. При прохождении проверки: `counted_ids.add(track_id)`, `class_counts[class_name] += 1`, `total_count += 1`

**Формат выходных данных:**
```python
counts: Dict[str, int]
# {"person": 5, "car": 2, "bicycle": 1}
```

**Критичные моменты:**

- **Двойной подсчёт невозможен** — `counted_ids` — это `Set`, каждый ID входит ровно раз.
- **Потеря трека → потеря счёта:** если трекер потерял объект до того, как тот вошёл в зону, счёт не произойдёт.
- **Переназначение ID:** если объект вышел из кадра и вернулся с новым `track_id` — он будет посчитан повторно. Это известное ограничение centroid-трекера без ReID.
- **count_zone = None:** без зоны каждый новый трек считается немедленно при первом появлении.

---

## Этап 5: Renderer → аннотированный кадр

**Что происходит:**
- Условная отрисовка по флагам из конфига (`show_detections`, `show_tracking_ids`, `show_counts`)
- Модификация кадра in-place (все операции пишут в переданный `np.ndarray`)
- Финальный `cv2.imshow()` — отображение в окне

**Порядок отрисовки:**
1. `render_detections` — тонкие рамки детекций (если включено)
2. `render_tracks` — жирные рамки с ID (если включено)
3. `render_counts` — панель в верхнем левом углу (если включено)
4. `render_fps` — FPS в верхнем правом углу (всегда)

**Критичные моменты:**
- Renderer не делает копию кадра — пишет поверх оригинала. Если нужен чистый кадр для записи/анализа, копию нужно делать до вызова Renderer.
- `cv2.addWeighted` для полупрозрачного фона панели счётчиков — лёгкая операция, но выполняется каждый кадр.

## Влияние на производительность

| Фактор | Влияние на FPS | Влияние на точность |
|--------|---------------|-------------------|
| `detector.confidence_threshold` | Нет | Высокий → пропуски; низкий → ложные треки |
| `detector.model` (yolov8n vs yolov8s) | Существенное | n быстрее, s точнее |
| `video.frame_width/height` | Умеренное | Высокое разрешение → лучше детекция мелких объектов |
| `tracker.max_distance` | Нет | Большой → смешение объектов; малый → частые потери |
| `tracker.max_disappeared` | Нет | Большой → стабильные треки; малый → преждевременное удаление |
| `display.show_detections` | Незначительное | Нет |
| `display.show_tracking_ids` | Незначительное | Нет |

---

## Влияние на стабильность

**Основные источники нестабильности:**
1. **Нестабильный инференс YOLO** — объект то обнаруживается, то нет → мерцание bbox, потенциальная смена ID
2. **Высокая скорость объектов** относительно FPS → превышение `max_distance`, потеря трека
3. **Окклюзии** — объекты перекрывают друг друга → неверное сопоставление центроидов
4. **Изменение класса объекта** — YOLO нестабильно классифицирует объект на границе confidence → в `class_counts` могут появиться счётчики разных классов для одного физического объекта

**Рекомендации для повышения стабильности:**
- Использовать `max_disappeared = 30–50` кадров (буфер на временные окклюзии)
- Не занижать `confidence_threshold` ниже 0.4 без необходимости
- При быстрых объектах увеличить `max_distance` или снизить разрешение для повышения FPS
